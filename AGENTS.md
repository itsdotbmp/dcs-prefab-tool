**Read the AGENTS.md for the area you're touching before writing code:**

| You're working on... | Read |
|---|---|
| In-DCS Lua framework (`sms.*`) | [`framework/AGENTS.md`](framework/AGENTS.md) |

## 1. Repo layout at a glance

- **`framework/`** — in-DCS Lua scripting framework. Runs inside the mission environment. One file per `sms.*` module. See `framework/AGENTS.md`.
- **`tools/cmd/dcs-sms/`** — Go source for `dcs-sms.exe`. Installs the hook + ME-mod, drives running missions and the Mission Editor via the filesystem mailbox. See `tools/cmd/dcs-sms/AGENTS.md`.
- **`tools/me-mod/`** — Lua source for the Mission Editor extension. Hosts the Prefab Manager UI and `verbs.lua` (the ME-side dispatch for `dcs-sms me <noun> <verb>`). Embedded into `dcs-sms.exe` via `go:embed`. See `tools/me-mod/AGENTS.md`.
- **`tools/lua/dcs-sms-hook.lua`** — the `Scripts/Hooks` Lua. Runs in DCS's unsandboxed hook environment. Polls the mailbox; executes `target=mission` requests during a running sim.
- **`tools/internal/`** — Go internal packages (`mailbox`, `hookstatus`, `proto`, `aiskill`, `dcspath`).
- **`docs/api/`** — per-module framework reference. Hand-authored. Load-bearing for users.
- **`docs/cli/`** — per-command CLI reference. **Autogenerated** by `dcs-sms doc`; do not hand-edit.
- **`docs/superpowers/specs/`** — design docs.

## 2. Three Lua environments

The framework, the CLI bridge, and the ME-mod live in three different Lua contexts. Don't conflate them — each has different sandbox rules and different globals.

| Environment | Sandbox | Used by |
|---|---|---|
| **Mission** | `os/io/lfs` nilled, `print` silent, Lua 5.1 | Framework (`framework/*.lua`); `dcs-sms exec --target mission` |
| **Hook** | Full I/O, LuaSocket, `lfs` | `tools/lua/dcs-sms-hook.lua` (`Scripts/Hooks/`) |
| **Mission Editor** | Full I/O, `lfs`/`io`; `UpdateManager` per-frame tick | `tools/me-mod/lua/...`; `dcs-sms exec --target gui`; every `dcs-sms me ...` verb |

The CLI bridges all three via the filesystem mailbox at `<Saved Games>/DCS/dcs-sms/{inbox,outbox,state,log}/`. The hook handles `target=mission` requests; the ME-mod's `bridge.lua` handles `target=gui` requests.

## 3. Documentation sync rule

The repo carries four "contributor reference" surfaces that **must not drift from the code**:

| Surface | What it is | When you must update it |
|---|---|---|
| `framework/AGENTS.md` §7 module index | One line per public `sms.*` module | When you add, remove, or rename a `sms.*` module |
| `docs/api/<module>.md` | Per-symbol framework reference + examples | When you add, remove, or rename any public `sms.*` symbol |
| `docs/cli/` | Per-command CLI reference | Run `dcs-sms doc` when adding/removing/renaming a CLI verb or flag. The `cli-docs-fresh` CI workflow enforces this. |
| `tools/me-mod/AGENTS.md` verb index | One line per `me <noun> <verb>` | When you add, remove, or rename a verb |

This rule applies at every stage:
- **Specs** must list the doc updates in their scope.
- **Plans** must include a concrete doc-update task.
- **Implementation** must land the doc update in the same PR / commit-set.
- **Review** treats a missing doc update as a blocker for any change touching public surface.

A PR that ships new public surface without the corresponding doc update is incomplete.

## 4. Versioning and releases

Two parallel semver-`0.x.y` tracks. Major (1.0) is reserved for the moment the public surface stops moving — explicitly not on the v1 roadmap.

| Component | Tag prefix | Canonical version |
|---|---|---|
| Framework | `framework-v0.X.Y` | `sms.version` in `framework/sms.lua` |
| ME-mod (+ `dcs-sms.exe`) | `me-mod-v0.X.Y` | `tools/me-mod/lua/dcs_sms_me/version.lua` |
| Prefab data format | `PREFAB_VERSION` (no git tag) | `framework/prefab_distill.lua` |

Bump rules under 0.x:
- **Patch** (`0.x.y` → `0.x.y+1`) — pure bug fix. No new public symbols, no behaviour change for working callers.
- **Minor** (`0.x.y` → `0.x+1.0`) — anything else: new public function / module / UI feature, OR a breaking change to an existing one (allowed under 0.x).

Workflow:
1. **In-source first.** Bump the version string in the same commit that lands the change. The git tag is the announcement; the in-source string is the truth.
2. **Annotated tags only** (`git tag -a`), prefixed by component (`framework-vX.Y.Z`, `me-mod-vX.Y.Z`). The tag annotation message is the release summary.
3. **Update `CHANGELOG.md`** in the same commit — section per component, newest entry on top.
4. **Push with `--follow-tags`** so the tag reaches `origin`.

Specs and plans that touch public surface call out the version bump as part of their scope, parallel to the doc-sync rule above.

## 5. Commit and PR style

- **Conventional commits.** `feat(<scope>)`, `fix(<scope>)`, `refactor(<scope>)`, `docs(<scope>)`, `chore(<scope>)`, `test(<scope>)`.
- **Scope follows the directory.** `framework`, `unit`, `events`, `me`, `me-mod`, `tools`, `bridge`, `hook`, `spec`, `plan`.
- **Short subject** (one line, ≤72 chars). Detail belongs in the linked spec, not the commit body.
- **Logical grouping over file-counting.** One commit per logical unit of work — not one per file, not bundles of unrelated changes.
- See `git log --oneline` on `main` for live examples; recent commits are the source of truth.

---

## 6. Worktrees

Use `.worktrees/` for feature branches that you want to develop in parallel without disturbing `main`'s working tree. The directory is gitignored. Standard procedure is in the `using-git-worktrees` skill.

---

## 7. Don't suggest commits silently

After substantial work, proactively offer to commit. **Never run `git commit` or `git push` without explicit confirmation.** Never push to remote without an explicit ask.

This mirrors the user's global preference in `~/.claude/CLAUDE.md`. It applies here too.
