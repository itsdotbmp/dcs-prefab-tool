# tools/cmd/dcs-sms/AGENTS.md — host-side CLI reference for AI agents

Contributor doc for the Go source under `tools/cmd/dcs-sms/`. The build produces `dcs-sms.exe` — the single binary that installs everything, drives running missions via the hook bridge, and drives the Mission Editor via the gui bridge.

> **Working elsewhere?** See [`../../AGENTS.md`](../../AGENTS.md) for repo orientation and cross-cutting rules. For the in-DCS framework see [`../../framework/AGENTS.md`](../../framework/AGENTS.md); for the ME-mod runtime and the `me <noun> <verb>` Lua half see [`../me-mod/AGENTS.md`](../me-mod/AGENTS.md).
>
> **User-facing CLI reference** lives at [`../../docs/cli/`](../../docs/cli/) and is **autogenerated** by `dcs-sms doc`. Don't hand-edit those files. This document is for *contributing code to the CLI*, not for using it.

---

## 1. Audience

You're adding, modifying, or fixing one of:

- A **top-level subcommand** (`dcs-sms exec`, `dcs-sms status`, `dcs-sms install-hook`, `dcs-sms install-me-mod`, `dcs-sms update`, `dcs-sms screenshot`, `dcs-sms doc`, `dcs-sms gen-units`, `dcs-sms install-ai-skill` / `uninstall-ai-skill`, `dcs-sms tail-log`).
- A **`me <noun> <verb>`** verb (the Go half). The Lua half lives in `tools/me-mod/lua/dcs_sms_me/verbs.lua` and is documented in [`../me-mod/AGENTS.md`](../me-mod/AGENTS.md).
- The **mailbox / bridge protocol** (`tools/internal/mailbox`, `tools/internal/proto`, `tools/internal/hookstatus`).
- The **installer logic** (`install_me_mod.go`, `installhook.go`, `install_ai_skill.go`, `update.go`).
- The **interactive menu** that runs when the .exe is double-clicked (`menu.go`).
- The **doc autogenerator** (`doc.go`).

## 2. File layout

The CLI is one Go package (`package main`). Each subcommand lives in its own `.go` file and registers itself in an `init()` function.

| File / pattern | Purpose |
|---|---|
| `main.go` | Entry point. Reads args, calls `dispatch`. |
| `dispatch.go` | `cmdInfo` registry type, `register` / `registerInfo` / `flagsOnly`, top-level dispatch, `printUsage`. |
| `menu.go` + `menu_test.go` | Double-click interactive menu (install / uninstall / update / set DCS path / install AI skill). Driven by TTY detection in `dispatch`. |
| `exec.go`, `status.go`, `taillog.go` | Bridge subcommands (talk to the running mission via the mailbox). |
| `installhook.go` | `install-hook` — copies `tools/lua/dcs-sms-hook.lua` into `<Saved Games>/DCS*/Scripts/Hooks/`. |
| `install_me_mod.go`, `uninstall_me_mod.go` | Mission Editor mod install/uninstall (patches `MissionEditor.lua`, copies the embedded `dcs_sms_me/` tree). |
| `install_ai_skill.go`, `uninstall_ai_skill.go` | Write/remove the embedded `dcs-sms` skill into `~/.claude` / `~/.agents` / `~/.gemini`. |
| `update.go` | Self-update via GitHub Releases API. |
| `screenshot_windows.go` | Windows-only screenshot via Win32 `PrintWindow`. |
| `genunits.go` | Regenerate `framework/constants/{units,statics}.lua` from `dcs-lua-datamine`. |
| `doc.go`, `doc_test.go` | `dcs-sms doc` — emits per-command markdown pages + index into `docs/cli/`. |
| `me.go` | The `me` dispatcher: noun → verb → handler routing, `runMeVerb` helper, `emitMeResponse`. |
| `me_<noun>_<verb>.go` | One file per `me <noun> <verb>` pair. Each one registers via `registerMeInfo` in an `init()`. |
| `me_color.go`, `me_resources_args.go`, `me_trigger_args.go` | Shared parsers for verb flags. |

Internal Go packages live at `../../internal/`:

| Package | Purpose |
|---|---|
| `tools/internal/mailbox` | Filesystem mailbox at `<Saved Games>/DCS/dcs-sms/{inbox,outbox,state}/` — request ID generation, atomic write, sweep. |
| `tools/internal/proto` | Wire format: `ExecRequest`, `ExecResponse`, `HookState` (heartbeat). |
| `tools/internal/hookstatus` | Reads heartbeats from `state/hook.json` (mission side) and `state/me.json` (ME side); merges them; routes `--target auto` based on what's alive. |
| `tools/internal/aiskill` | Embedded SKILL.md + per-agent paths resolver. |
| `tools/internal/dcspath` | DCS install-path auto-detection + user-paste sanitization. |

## 3. The cmdInfo registry pattern

Every subcommand registers itself in an `init()` function so `dispatch` can route to it and `dcs-sms doc` can introspect it.

```go
func init() {
    registerInfo("status", cmdInfo{
        Run:      statusCmd,
        Flags:    flagsOnly(statusFlags),
        Synopsis: "report whether the hook is alive and a mission is loaded",
    })
}
```

`cmdInfo` is defined in `dispatch.go`:

```go
type cmdInfo struct {
    Run      commandFunc          // (args, stdout, stderr) → exitCode
    Flags    func() *flag.FlagSet // for doc generator; nil = hidden
    Synopsis string               // one-line — used in `printUsage` and docs/cli/
    Examples []string             // TODO(#49): consumed by doc gen
}
```

### Rules

- **`Run` is required.** Everything else is optional.
- **`Synopsis` and `Flags` together gate doc generation.** A command with both set appears in `docs/cli/<slug>.md`. Either missing → the command is hidden by `dcs-sms doc` (suitable for internal/legacy commands).
- **`flagsOnly` is the bridge** from your typed-opts builder (`func() (*flag.FlagSet, *fooOpts)`) to the `func() *flag.FlagSet` shape `cmdInfo.Flags` expects. Use it — don't hand-write a second flags-only function.
- **Pure functions over `os.*`.** Subcommand handlers take `(args, stdout, stderr)` and return an `int` exit code. They must not call `os.Exit` or read from `os.Stdin` directly — `main.go` does that wiring. This is what makes the tests in `*_test.go` straightforward.

### Subcommand-authoring template

```go
func init() {
    registerInfo("foo", cmdInfo{
        Run:      fooCmd,
        Flags:    flagsOnly(fooFlags),
        Synopsis: "one-line description for `dcs-sms --help` and docs/cli/",
    })
}

type fooOpts struct {
    Timeout time.Duration
    Pretty  bool
}

func fooFlags() (*flag.FlagSet, *fooOpts) {
    opts := &fooOpts{}
    fs := flag.NewFlagSet("foo", flag.ContinueOnError)
    fs.DurationVar(&opts.Timeout, "timeout", 30*time.Second, "wall-clock timeout")
    fs.BoolVar(&opts.Pretty, "pretty", false, "indent JSON output")
    return fs, opts
}

func fooCmd(args []string, stdout, stderr io.Writer) int {
    fs, opts := fooFlags()
    fs.SetOutput(stderr)
    if err := fs.Parse(args); err != nil {
        return 2
    }
    // ... do the work, write to stdout/stderr, return exit code ...
    _ = opts
    return 0
}
```

Add the file. The `init()` registers it. No other wiring needed.

## 4. Exit codes (conventional)

| Code | Meaning |
|---|---|
| `0` | Success. |
| `1` | The command ran end-to-end but the operation it performed reports failure (e.g. a verb returned `{ok=false}`, an exec snippet raised). |
| `2` | Bad CLI usage — unknown command, missing required flag, parse failure. |
| `3` | Environment/setup failure — couldn't find Saved Games, couldn't write to the mailbox, install path missing. |
| `4` | Bridge target unavailable — e.g. `--target gui` requested but the ME-side bridge is disabled. |

The doc generator surfaces these in the `--help` output of each command. Stay consistent.

## 5. The bridge protocol

The CLI talks to in-DCS Lua over a filesystem mailbox at `<Saved Games>/DCS/dcs-sms/`:

```
inbox/   *.req.json    — requests written by the CLI
outbox/  *.res.json    — responses written by the hook (target=mission) or ME-mod (target=gui)
state/   hook.json     — heartbeat from the Scripts/Hooks side
state/   me.json       — heartbeat from the ME-mod side
log/                   — bridge-side logs
```

### Request shape (`proto.ExecRequest`)

```json
{
  "id":         "<random ID>",
  "kind":       "exec",
  "target":     "mission" | "gui",
  "code":       "<lua source>",
  "timeout_ms": 30000,
  "created_at": "RFC3339Nano UTC"
}
```

### Response shape (`proto.ExecResponse`)

```json
{
  "id":             "<matches request>",
  "ok":             true | false,
  "output":         "<captured print output>",
  "return_value":   <verb-specific JSON>,
  "error":          { "message": "...", "traceback": "..." } | null,
  "frame_executed": <sim frame number>,
  "duration_ms":    <number>
}
```

### Target routing

- `target=mission` — handled by `tools/lua/dcs-sms-hook.lua`. Requires a running mission. Sandboxed Lua env.
- `target=gui` — handled by `tools/me-mod/lua/dcs_sms_me/bridge.lua`. Requires the Mission Editor to be open AND the "External execution" toggle to be ON (`_G.DCS_SMS_GUI_BRIDGE_ENABLED == true`). Full-IO Lua env, can read/write the live `mission` table.
- `target=auto` — `hookstatus.RouteForTarget` picks the live side based on the two heartbeats. Falls back to error exit 4 if neither is available.

`me <noun> <verb>` always routes to `gui`. The `exec` subcommand exposes the choice via `--target`.

## 6. Authoring a `me <noun> <verb>` — Go half

Every `me <noun> <verb>` command is a two-file pair: a Go file here (the CLI surface, flag parsing, doc autogen entry) and a Lua function in `tools/me-mod/lua/dcs_sms_me/verbs.lua` (the actual ME-side work).

### Template (Go side)

```go
package main

import (
    "flag"
    "fmt"
    "io"
    "time"
)

func init() {
    registerMeInfo("widget", "set-foo", cmdInfo{
        Run:      meWidgetSetFooCmd,
        Flags:    flagsOnly(meWidgetSetFooFlags),
        Synopsis: "set a widget's foo property to the given value",
    })
}

type meWidgetSetFooOpts struct {
    Name       string
    Foo        string
    Timeout    time.Duration
    Pretty     bool
    SavedGames string
}

func meWidgetSetFooFlags() (*flag.FlagSet, *meWidgetSetFooOpts) {
    opts := &meWidgetSetFooOpts{}
    fs := flag.NewFlagSet("me widget set-foo", flag.ContinueOnError)
    fs.StringVar(&opts.Name, "name", "", "widget name (required)")
    fs.StringVar(&opts.Foo, "foo", "", "new value for foo (required)")
    fs.DurationVar(&opts.Timeout, "timeout", 30*time.Second, "wall-clock timeout")
    fs.BoolVar(&opts.Pretty, "pretty", false, "indent JSON output")
    fs.StringVar(&opts.SavedGames, "saved-games", "", "override Saved Games path")
    return fs, opts
}

func meWidgetSetFooCmd(args []string, stdout, stderr io.Writer) int {
    fs, opts := meWidgetSetFooFlags()
    fs.SetOutput(stderr)
    if err := fs.Parse(args); err != nil {
        return 2
    }
    if opts.Name == "" || opts.Foo == "" {
        fmt.Fprintln(stderr, "dcs-sms me widget set-foo: --name and --foo are required")
        return 2
    }

    // Build the Lua args table as a literal expression. Keep it simple —
    // `runMeVerb` wraps this into `return require('dcs_sms_me.verbs').<name>(<args>)`.
    luaArgs := fmt.Sprintf("{ name = %q, foo = %q }", opts.Name, opts.Foo)
    resp, exitCode := runMeVerb("widget_set_foo", luaArgs, opts.Timeout, opts.SavedGames, stderr)
    if exitCode != 0 {
        return exitCode
    }
    return emitMeResponse(resp, opts.Pretty, stdout)
}
```

### `runMeVerb` does the heavy lifting

`me.go` exposes two helpers your handler should always use:

- **`runMeVerb(verb, luaArgsExpr, timeout, savedGames, stderr)`** — wraps `luaArgsExpr` as `return require('dcs_sms_me.verbs').<verb>(<args>)`, writes the request, waits for the response, returns `(proto.ExecResponse, exitCode)`. Returns exit code `3` for environment errors, `4` if the gui bridge is disabled, `2` for poll timeout.
- **`emitMeResponse(resp, pretty, stdout)`** — JSON-encodes the response and emits to stdout. Returns `0` if `resp.OK` is true, `1` otherwise.

### Building the Lua args table

Keep it a literal Lua expression. For typical shapes:

- Strings: `%q` (Go's `%q` produces a `"..."` string with escapes — also valid Lua).
- Numbers: `%v` or `%d` / `%g`.
- Bools: `%v` works for `true`/`false`.
- Lists / nested tables: build them up. Some verbs already have shared parsers in `me_color.go`, `me_resources_args.go`, `me_trigger_args.go`.

If a flag is optional, omit it from the Lua table when unset — don't pass `nil`. The Lua verb side reads `args.foo` and gets `nil` when the key is absent, same as if you'd written it.

### Add tests

Each verb pair gets a `_test.go` file that exercises flag parsing and the args-to-Lua mapping. Don't dial DCS in tests — use the pure-function shape (`(args, stdout, stderr) → exit`) and assert on the captured stdout/stderr.

### Don't forget the Lua half

Adding a Go file alone won't work. The Lua function must exist in `tools/me-mod/lua/dcs_sms_me/verbs.lua` with the matching snake_case name (`widget_set_foo`). See [`../me-mod/AGENTS.md`](../me-mod/AGENTS.md) for the Lua-side authoring guide.

### Regenerate docs

After landing a new verb:

```sh
dcs-sms doc                         # regenerates docs/cli/
```

The `cli-docs-fresh` CI workflow gates PRs on this — a stale `docs/cli/` directory fails CI.

## 7. Doc autogenerator (`dcs-sms doc`)

`doc.go` walks the `commands` and `meCommands` registries, pulls each `cmdInfo`'s `Synopsis` + `Flags`, and renders one markdown page per command into `docs/cli/<slug>.md` plus an index at `docs/cli/README.md`.

When you change a command:
- **Edit the `Synopsis` field** to update the one-line description.
- **Edit the `Flags` builder** to add/remove/rename flags. Each flag's `Usage` string becomes the description in the generated page.
- **Run `dcs-sms doc`** locally and commit the regenerated `docs/cli/`. CI rejects PRs that didn't.

The synopsis appears in three places: `dcs-sms --help`, the `me` group of `dcs-sms me --help`, and the generated `docs/cli/<slug>.md` heading. Keep it ≤80 chars and active-voice.

## 8. The required `MissionScripting.lua` edit

`exec --target mission`, `status`, and `tail-log` all rely on the `Scripts/Hooks` Lua bypassing DCS's mailbox sandbox. The user must edit `<DCS install>/Scripts/MissionScripting.lua` once to comment out the `sanitizeModule('os')` / `('io')` / `('lfs')` lines (the same edit `dcs_code_injector` requires).

This is documented in `tools/cmd/dcs-sms/README.md` and is **not** something the installer can do automatically (and shouldn't — DCS itself rewrites this file on update). Reference it in user-facing error messages when bridge calls fail, don't try to patch it.

`install-me-mod` does **not** need this edit. The ME-side bridge runs in the ME Lua state where I/O isn't sandboxed.

## 9. Build, install, run

From the repo root:

```sh
cd tools
go build ./cmd/dcs-sms              # produces tools/dcs-sms[.exe]
go test  ./cmd/dcs-sms              # run unit tests
go test  ./...                      # all Go tests, including internal/
```

After editing **anything under `tools/me-mod/lua/`**, you must rebuild `dcs-sms.exe` *and* reinstall it into DCS — the Lua tree is `go:embed`ded into the binary. Just rebuilding isn't enough; the user-installed copy under `<DCS>/MissionEditor/modules/dcs_sms_me/` is stale until you run `dcs-sms install-me-mod` again.

Self-update for end users is `dcs-sms update`; contributors building from source skip it.

## 10. When you write new CLI code

- **Mirror the `cmdInfo` pattern.** Don't add a new dispatch path; use `registerInfo` / `registerMeInfo` and a `flagsOnly`-wrapped flags builder.
- **Pure functions for handlers.** `func(args []string, stdout, stderr io.Writer) int`. No `os.Exit`. No `os.Stdin` reads outside `main`.
- **Use the internal packages.** Don't reach into `<Saved Games>` directly — use `mailbox.New` / `mailbox.NewID` / `mailbox.WriteRequest` / `hookstatus.ReadMerged` / `dcspath.Detect`.
- **Set exit codes consistently** (see §4).
- **Write a `_test.go`** with at least one happy path and one flag-parse failure case. The pure-function shape makes this cheap.
- **Update the synopsis** when behavior changes meaningfully. Regenerate `docs/cli/` and commit.
- **Bump the ME-mod version** if your change is user-visible behavior (the CLI ships under the `me-mod-v*` track — see [`../../AGENTS.md`](../../AGENTS.md#4-versioning-and-releases)). Update `CHANGELOG.md` in the same commit.

Cross-cutting rules (commit style, versioning, doc-sync at the repo level) live in [`../../AGENTS.md`](../../AGENTS.md).
