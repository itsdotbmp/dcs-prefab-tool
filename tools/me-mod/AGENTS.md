# Part 1 — Using the verbs

## 1.1 The big picture

`dcs-sms.exe` is a host-side CLI. The ME-mod is a Lua module DCS loads when the Mission Editor starts. They communicate through a filesystem mailbox in `<Saved Games>/DCS/dcs-sms/`. When you run

```sh
dcs-sms me unit set-pos --name "Hornet-1" --north 12000 --east -3500
```

the CLI writes a JSON request to `inbox/`, the ME-mod's `bridge.lua` (polling on every Mission Editor tick) loads and executes the Lua, writes the response to `outbox/`, and the CLI reads it and prints it. **No mission needs to be running** — the verbs operate on the *editable* mission table that the Mission Editor exposes, so you can do everything you'd do with the GUI without launching a sim.

## 1.2 Prerequisites

Once-only setup:

1. The user installed the ME-mod (`dcs-sms install-me-mod`, double-click menu option 1, or OVGME).
2. The user opened DCS, opened the Mission Editor, and clicked **DCS-SMS → External execution: OFF** to flip it ON. This sets `_G.DCS_SMS_GUI_BRIDGE_ENABLED = true` in the ME's Lua state. The toggle is session-only (default OFF at every DCS launch — no persistence) and gates *only* `target=gui` requests.

If the user hasn't toggled it on, every `me <verb>` call returns exit code **4** with a clear error pointing at the menu item. Tell the user to flip the toggle; don't try to bypass it.

## 1.3 Discovery

```sh
dcs-sms me --help                         # lists every noun + verb pair
dcs-sms me <noun> <verb> --help           # flags for one specific verb
dcs-sms doc                               # (in the repo) regenerates docs/cli/
```

The autogenerated per-verb reference is at [`../../docs/cli/`](../../docs/cli/) — one markdown page per command (`me-unit-set-pos.md`, `me-trigger-create.md`, etc.). Each page lists every flag with type, default, and description.

## 1.5 Coordinate convention — read this before placing anything

**All public verbs that take a map position use the flag triplet `--north`, `--east`, `--alt`** (all in meters, theatre-origin-relative). DCS exposes positions in two contradictory forms — both abbreviate to "x/y/z" but with different meanings:

- **Mission table** (the `.miz` file format, what the ME persists): `{x = north, y = east}` — `y` means east here, no `z`, altitude is a separate `alt` field.
- **Runtime 3D engine** (vec3 in mission-env scripting): `{x = north, y = altitude, z = east}` — `y` means altitude here, `z` means east.

The CLI flag names (`--north` / `--east` / `--alt`) are unambiguous and translate to whichever shape the underlying API needs. **Never** pass `--x` / `--y` to a `me ...` verb — those don't exist; the ambiguity is the whole reason the flag names are what they are.

Headings and rotations are **in degrees**, 0=north, 90=east, clockwise. Altitudes are **in meters**.

See [`research/me-bridge-discovery-2026-05-08.md`](../../research/me-bridge-discovery-2026-05-08.md) → "DCS ME coordinate axes (critical correction)" for the discovery story.

## 1.6 Output and failure mode

Every `me <verb>` emits one JSON object to stdout. The shape is `proto.ExecResponse`:

```json
{
  "id": "<request id>",
  "ok": true,
  "output": "",
  "return_value": { "ok": true, /* verb-specific fields */ },
  "error": null,
  "frame_executed": 12345,
  "duration_ms": 3
}
```

When parsing, the field you want is usually `return_value` — that's the verb's actual payload. `return_value.ok` mirrors the outer `ok` for unified handling.

Add `--pretty` for indented JSON; useful while exploring, omit in scripts.

**Exit codes:**

| Code | Meaning |
|---|---|
| `0` | The verb returned `{ok=true, ...}`. |
| `1` | The verb returned `{ok=false, error="..."}` — the operation failed, but the request reached the ME (a unit name didn't exist, a flag value was invalid, the ME refused the mutation, etc.). |
| `2` | Bad CLI usage (missing required flag, parse failure). |
| `3` | Environment failure — couldn't find Saved Games, couldn't write to the mailbox. |
| `4` | The gui bridge isn't enabled. Tell the user to flip **DCS-SMS → External execution: OFF → ON** in the ME. |

When you call a verb, check exit code first; if `0`, parse the response. **Don't retry on exit 1** — the ME explicitly rejected the request, retrying won't help. Read `return_value.error` to see what to fix.

## 1.7 Common gotchas

- **A mission must be open** for most verbs that read or mutate the mission table. `file new --map <theatre>` and `file open --path <path>` get you to a usable state; the editor refuses verbs that depend on a mission table when nothing is loaded.
- **`save` doesn't auto-`save-as`.** If the mission has never been saved, `file save` errors — use `file save-as --path <path>` for the first write.
- **The ME caches map labels per-group.** Bulk unit-level position mutations don't always refresh the F10 map until you `group:set-pos` the parent group too. `verbs.lua` does this for you on the verbs that need it — but if you're chaining `exec --target gui '<raw lua>'`, remember the `refresh_group_view` pattern.
- **`set-late-activation`, `set-uncontrolled`** apply at the group level even though they read like per-unit flags. That matches the .miz schema.
- **Airbase verbs only return data for fixed airfields**, not ship-deck/FARP statics. Carriers and FARPs are *static objects with parking* rather than airbase records — the data lives in a different place. Filter accordingly.
- **Trigger predicates are descriptor-driven.** Use `trigger list-predicates` to discover available conditions and `trigger describe-predicate --name <p>` to see its parameter shape before constructing one.

## 1.8 Running raw Lua against the editor

For things no verb covers, drop to:

```sh
dcs-sms exec --target gui --code '<lua>'
```

This runs your Lua in the same ME state the verbs do — full I/O, full access to the editable `mission` table. **Flag the gap.** If you find yourself doing this for anything other than one-shot exploration, the right answer is usually a new verb. See [`Part 2.5`](#25-adding-a-new-verb-end-to-end) for the workflow, or ask the user to file an issue.

# Part 2 — Contributing to the ME-mod

## 2.1 The ME Lua environment

The Mission Editor's Lua state is *not* the mission scripting sandbox. You have:

- **Full `io` / `lfs` / `os`** — the bridge uses these to poll inbox files and write outbox responses.
- **Lua 5.1**, same as the mission env.
- **`require()`** with `package.loaded` — modules persist across mission opens.
- **`UpdateManager`** — DCS's GUI per-frame tick. `bridge.lua` uses `UpdateManager.add(tick)` to poll the inbox.
- **`log.write(tag, level, msg)`** — DCS's global logger. Use it; `print` goes to `dcs.log` but is harder to grep for.
- **The live `mission` table** — `require('me_mission').mission` is the same table the ME's GUI mutates when the user clicks buttons. Mutating it from a verb is exactly equivalent to the user clicking.

You do **not** have:

- The `sms.*` framework. That only loads when a mission runs; the ME isn't a mission. Verbs work directly against DCS's editor internals.
- `env.info` / `env.warning` / `env.error`. Those are mission-env globals. Use `log.write` instead.

## 2.3 The embed workflow

`tools/me-mod/lua/` is `go:embed`ed into `dcs-sms.exe`. When you edit any Lua file in this tree:

1. **Rebuild the binary.** `cd tools && go build ./cmd/dcs-sms` — produces `tools/dcs-sms.exe`. The Lua changes don't reach the user's DCS install yet.
2. **Reinstall.** `dcs-sms install-me-mod` — copies the embedded Lua tree to `<DCS install>/MissionEditor/modules/dcs_sms_me/`, overwriting the previous copy. The `MissionEditor.lua` patch line stays put.
3. **Restart DCS.** Lua under `MissionEditor.lua` loads once at DCS start; you cannot hot-reload without a full restart. (Dev-reload via `Ctrl+Shift+R` inside the ME *partially* reloads things but does not re-run `MissionEditor.lua` — don't rely on it.)

A common contributor mistake: editing Lua, rebuilding, and forgetting to reinstall. The installed copy under `<DCS>/MissionEditor/modules/` is what runs; the source tree is dead until reinstalled.

## 2.4 Failure model for verbs

Every verb in `verbs.lua` returns a uniform table:

```lua
function M.widget_set_foo(args)
  -- ...do the work...
  if not args.name or args.name == '' then
    return { ok = false, error = 'name is required' }
  end
  -- ...mutation...
  return { ok = true, name = args.name, foo = args.foo }
end
```

- **`ok = true`** on success, with verb-specific extra fields (the mutated entity's id, the new value, whatever the caller will want).
- **`ok = false, error = "..."`** on failure, with a single human-readable string. The CLI surfaces this verbatim.
- **Never `error()` or `assert()`.** If something throws inside your verb, the bridge's `xpcall` catches it and emits exit code 1 with a traceback, but the user gets a worse experience. Wrap risky ME-internal calls in `pcall` and return `{ok=false, error=...}` yourself.

This contract is similar in spirit to the framework's "log + nil + never throw", but the shape is different because the verb's return value is the wire response. Don't use `nil` to signal failure — the bridge would JSON-encode it as `null` and the CLI would treat that as missing-payload.

`verbs.lua` already has a uniform pcall pattern wrapping each verb body — read a few existing verbs (`verbs.unit_set_pos`, `verbs.trigger_create`) to see the conventional shape before writing a new one.

## 2.5 Adding a new verb end-to-end

A verb is a pair: a function in `verbs.lua` (does the work) and a Go file in `tools/cmd/dcs-sms/` (the CLI surface + flag parsing). You must add both.

### Step 1 — write the Lua

In `tools/me-mod/lua/dcs_sms_me/verbs.lua`, add a function whose name matches the noun-verb pair you intend to register (snake_case):

```lua
-- ============================================================
-- widget verbs
-- ============================================================

function M.widget_set_foo(args)
    if type(args) ~= 'table' then
        return { ok = false, error = 'args must be a table' }
    end
    if type(args.name) ~= 'string' or args.name == '' then
        return { ok = false, error = 'name is required' }
    end
    if type(args.foo) ~= 'string' then
        return { ok = false, error = 'foo is required' }
    end

    local widget = find_widget_in_mission(args.name)  -- (use existing helpers)
    if not widget then
        return { ok = false, error = 'widget not found: ' .. args.name }
    end

    widget.foo = args.foo

    -- If the change is visible in the editor, refresh the relevant panel.
    local ok, err = pcall(refresh_widget_panel, widget)
    if not ok then
        log.write('sms.me.verbs', log.WARNING, 'panel refresh failed: ' .. tostring(err))
    end

    return { ok = true, name = widget.name, foo = widget.foo }
end
```

**Style notes:**

- Validate inputs at the top; return early with `{ok=false, error=...}` for caller-error cases.
- Reuse the helpers already at the top of `verbs.lua` (`walk_groups`, `find_unit_in_mission`, `refresh_group_view`, `strip_back_refs`). Don't reinvent them.
- After any mutation that's visible in the ME UI, call the appropriate refresh helper (`refresh_group_view`, `panel_route.fixActions`, etc.) — otherwise the user sees stale state until they click something.
- Use `log.write('sms.me.verbs', log.INFO|WARNING|ERROR, msg)` for diagnostic logs. They land in `dcs.log` prefixed with the tag.

### Step 2 — write the Go side

See [`../cmd/dcs-sms/AGENTS.md`](../cmd/dcs-sms/AGENTS.md#6-authoring-a-me-noun-verb--go-half) for the full Go-side template. Summary:

```go
// tools/cmd/dcs-sms/me_widget_set_foo.go
func init() {
    registerMeInfo("widget", "set-foo", cmdInfo{
        Run:      meWidgetSetFooCmd,
        Flags:    flagsOnly(meWidgetSetFooFlags),
        Synopsis: "set widget.foo to the given value",
    })
}
// ... flag struct, flags builder, Cmd function that calls runMeVerb("widget_set_foo", ...) ...
```

The Lua function name (`widget_set_foo`) and the (noun, verb) pair (`"widget", "set-foo"`) are linked only by convention — Go translates `kebab-case` → `snake_case` in the Lua call. Stay consistent or the dispatcher won't find your function.

### Step 3 — test

Add `tools/cmd/dcs-sms/me_widget_set_foo_test.go` exercising flag parsing and the Lua-args expression. For pure-Lua verb logic, you can also add a Lua test under `tools/me-mod/test/` using the `mock_me_mission.lua` harness — see `test_verbs_route.lua` for the pattern.

### Step 4 — regenerate docs

```sh
cd tools && go build ./cmd/dcs-sms
./dcs-sms doc                       # writes docs/cli/me-widget-set-foo.md and refreshes docs/cli/README.md
```

CI's `cli-docs-fresh` workflow fails the PR if `docs/cli/` is out of date.

### Step 5 — update this file's verb index

Add `set-foo` to the `widget` row in [§1.4](#14-verb-namespace) above (or add a new row if you introduced a new noun). The doc-sync rule says: same commit as the code.

### Step 6 — bump the ME-mod version + CHANGELOG

Edit `tools/me-mod/lua/dcs_sms_me/version.lua` and add a CHANGELOG entry. Versioning rules are in [`../../AGENTS.md`](../../AGENTS.md#4-versioning-and-releases). A new verb is **minor**.

### Step 7 — rebuild and reinstall on your DCS

Rebuild `dcs-sms.exe`, run `dcs-sms install-me-mod`, restart DCS, open the ME, smoke-test the verb manually:

```sh
dcs-sms me widget set-foo --name "Test" --foo "bar" --pretty
```

If you don't have DCS running locally, the Go tests and the Lua mock tests are the floor — but a manual smoke before tagging a release is non-negotiable.

## 2.6 The `verbs.lua` structure

`verbs.lua` is one file (currently ~7000 lines) because every verb shares the same set of small helpers and the same JSON-table return shape. Don't split it into one-file-per-noun "for cleanliness" — the helpers at the top are tightly coupled to multiple verbs and you'd end up with circular requires.

Layout inside `verbs.lua`:

1. **Top comment block** — coordinate convention, error convention, conventions for new verbs.
2. **Shared helpers** — `walk_groups`, `strip_back_refs`, `refresh_group_view`, `find_unit_in_mission`, etc. Declared with `local` so Lua's forward-reference rules let any verb below them call them.
3. **Verbs by noun** — each noun has a `-- == noun verbs ==` banner comment, then all its verb functions. Order roughly mirrors the noun list in [§1.4](#14-verb-namespace).
4. **`return M`** at the bottom.

When adding a new verb, find the existing noun's section and add inside it (alphabetical by verb name is conventional but not strict). If the noun is brand new, add a new banner section and place it alphabetically among the others.

## 2.7 ME API quirks you'll hit

- **The mission table is the source of truth.** `require('me_mission').mission` returns the table the editor is currently editing. Most verbs walk it directly. The runtime DCS APIs (`Group.getByName`, `coalition.addGroup`) are **not** available here — those are mission-env only.
- **Panel refresh is needed after mutation.** If a panel (route, payload, options) is open when you mutate the underlying data, the panel won't redraw automatically. Use the canonical refresh path for the panel (`require('panel_route').fixActions` and friends — read existing verbs for the right call).
- **`mapObjects` lazy creation.** Disk-loaded groups don't have `mapObjects` until selected. After a position mutation, call `Mission.create_group_map_objects(g)` if `g.mapObjects == nil`, then `Mission.update_group_map_objects(g)`. `refresh_group_view(g)` in `verbs.lua` does this.
- **`dictionary.fixDict`** — trigger action text, comments, radio-text need to go through this so the dictionary serializer can find them. Inline literal strings get dropped silently.
- **Descriptor walking** — triggers and predicates are descriptor-driven. The descriptors live in `me_predicates.rulesDescr` / `actionsDescr`. Don't hardcode the param shapes; query the descriptor.
- **`isVisible()` over monkey-patch** — to detect "is the trigger panel currently showing this trigger", call the widget's `isVisible()` rather than tracking it externally.

The research notes in [`../../research/me-bridge-discovery-2026-05-08.md`](../../research/me-bridge-discovery-2026-05-08.md) and the route-geometry spec in [`../../docs/superpowers/specs/2026-05-11-me-route-geometry.md`](../../docs/superpowers/specs/2026-05-11-me-route-geometry.md) document the discovery process for several of these. Read them before reaching for monkey-patching.

## 2.9 The `sms_window` chrome (for new tool windows)

If you're adding a new floating window (Prefab Manager is the first; mass-rename tool, force-build tool, etc. are plausible follow-ups), don't roll your own chrome — use `sms_window`.

The handle provides:
- Branded title bar (`Coconut Cockpit · DCS-SMS — <name> v<version>`).
- Footer band: separator + colored status `Static` for `set_status` / `flash_status` / `clear_sticky_status`.
- Close button that **hides** (doesn't destroy) the window.
- Auto-hide on `File > New` / `File > Open`.
- `Ctrl+Z` wired to the project-wide undo bus.
- Resize with min-size clamp and footer reposition.

Usage:

```lua
local sms_window = require('dcs_sms_me.sms_window')

local W = { sms_window = nil }

function M.show()
    if W.sms_window then W.sms_window:show(); return end
    W.sms_window = sms_window.new({
        title    = 'My Tool',
        size     = { w = 400, h = 300 },
        min_size = { w = 320, h = 200 },
        on_resize = function(swin, x, y, w, h) ... end,
        -- on_undo, on_close also optional
    })
    if not W.sms_window then return end
    -- ... build body via W.sms_window:raw():insertWidget(...) ...
    W.sms_window:show()
end
```

Helpers on the module (call on the module, not the handle): `sms_window.compose_title(title, version)`, `sms_window.default_on_undo(handle)`. Severities for status: `'info'`, `'success'`, `'warning'`, `'error'`.

Design rationale: [`../../docs/superpowers/specs/2026-05-08-me-sms-window-base-class.md`](../../docs/superpowers/specs/2026-05-08-me-sms-window-base-class.md).

## 2.10 The menu, the toggle, the bootstrap chain

If you need to add a new menu item under **DCS-SMS** (e.g. opening your new tool window), the path is in `menu.lua`. Follow the existing `Prefab Manager` / `About` / `External execution` pattern:

1. Build a `menu:newItem('My Tool')`.
2. Copy the sibling-item skin (`sibling_menu.missionOptions:getSkin()` etc.).
3. Set `item.func = function() require('dcs_sms_me.my_tool').toggle() end`.

`menu.install` runs from `init.lua` and either adds the entry immediately (if `me_menubar.menuBar` is ready) or monkey-patches `me_menubar.show` so the entry appears on the first paint. This pattern works across DCS builds.

The **External execution** toggle flips `_G.DCS_SMS_GUI_BRIDGE_ENABLED`. Anything in the ME-mod that wants to react to bridge state should read this global directly — it's the source of truth.

## 2.11 When you write new ME-mod code

- **Always wrap** any vanilla ME API call you're not 100% sure about in `pcall`. ED rewrites internal APIs between builds; defensive pcalls keep the mod usable on future builds.
- **Log via `log.write('sms.me.<subsystem>', ...)`**. Useful tags: `sms.me`, `sms.me.bridge`, `sms.me.verbs`, `sms.me.prefab`. New subsystems should pick their own clear tag.
- **Never throw out of a verb.** Return `{ok=false, error=...}` instead.
- **Refresh after mutation.** If the panel/map UI shows what you changed, you need to refresh it. Use `refresh_group_view` and the panel-fixer helpers in `verbs.lua` rather than reinventing them.
- **Update this file's verb index** ([§1.4](#14-verb-namespace)) on every verb add/remove/rename. Same commit.
- **Update `docs/cli/`** by running `dcs-sms doc`. Same commit. CI fails the PR otherwise.
- **Bump `version.lua` and `CHANGELOG.md`.** Same commit. Rules in [`../../AGENTS.md`](../../AGENTS.md#4-versioning-and-releases).

Cross-cutting rules (commit style, versioning, doc-sync at the repo level) live in [`../../AGENTS.md`](../../AGENTS.md).
