# `sms.rule` — declarative trigger rules — design spec

**Status:** approved (brainstorming complete 2026-04-30)
**Branch:** `feat/sms-rule`
**Worktree:** `.worktrees/sms-rule`

## Goal

Replace the Mission Editor's trigger panel with code-defined rules. A rule is a named, polled, condition-action pair with a small lifecycle (`once` / `continuous` / `toggle`), optional fire cooldown and condition sustain, and a developer escape hatch for instant testing. Inspired by MOOSE's `RULE` class but reshaped to the dcs-sms idiom: per-rule timers instead of a shared gameloop, callable handles, options-table constructor, log + nil failure mode, no inheritance.

## User value

DCS's Mission Editor triggers are tedious to author and impossible to diff or refactor. MOOSE's `RULE` class solved the authoring problem but on top of a heavy `BASE` inheritance chain and a single shared scheduler that ticked every rule at the highest required frequency.

After this work the user gets:

- One file per behavioral rule, fully expressible in code:
  ```lua
  sms.rule("helicopter_height", {
    type          = sms.rule.TYPE.ONCE,
    interval      = 2,
    dev_condition = function() return MIZ.helicopter_height end,
    condition     = function() ... end,
    action        = function() ... end,
  })
  ```
- A `dev_condition` escape hatch that fires the action *immediately and unconditionally* when set true (bypasses sustain and cooldown), so testing actions doesn't require setting up real game state.
- Three lifecycle types that cover the patterns the user has in production today (`ONCE`, `CONTINUOUS`, `TOGGLE`), plus orthogonal `cooldown` and `sustain` knobs that compose with all of them.
- A registry (`sms.rule.get / .all / .remove / .test_all`) for runtime introspection and for diagnostic dry-runs of every rule's callbacks.

## Scope

### In scope

1. New module `framework/rule.lua` exposing `sms.rule` (callable constructor + module functions + `TYPE` enum table).
2. Three lifecycle types: `sms.rule.TYPE.ONCE`, `sms.rule.TYPE.CONTINUOUS`, `sms.rule.TYPE.TOGGLE`. Stored internally as the lowercase strings `"once"`, `"continuous"`, `"toggle"`.
3. Opts fields: `type` (required), `condition` (required), `action` (required), `interval` (default `1`, must be `> 0`), `cooldown` (default `0`, no cooldown), `sustain` (default `0`, no sustain), `dev_condition` (default no-op).
4. Per-rule timer via `sms.timer.every(interval, check)` — no shared scheduler, no manager-owned tick loop.
5. State machine that respects type semantics, cooldown gating, sustain accumulation, and `dev_condition` bypass. All callbacks invoked via `pcall`; throws are logged at `error` and never propagate.
6. Handle methods: `:get_name()`, `:get_type()`, `:is_active()`, `:fire()`, `:stop()`, `:reset()`, `:set_interval(sec)`, `:set_cooldown(sec)`, `:set_sustain(sec)`. Both method-style and module-style call forms, like every other entity module.
7. Module-level registry functions: `sms.rule.get(name)`, `sms.rule.all()`, `sms.rule.remove(name)`, `sms.rule.test_all()`.
8. Construction-time name collision behavior: existing rule with the same name is `:stop()`-ed and replaced; logged at `info`.
9. New API doc page `docs/api/rule.md` with full per-symbol reference, opts table, type semantics, and a worked example end-to-end (the helicopter_height rule).
10. `AGENTS.md` §7 module index gets a new line for `sms.rule`.
11. `framework/load_all.lua` updated to load `rule.lua` after `timer.lua`.
12. New smoke test `framework/test/smoke_rule.sh` covering bad-arg validation, each type's lifecycle, cooldown gating, sustain accumulation, dev_condition bypass, name collision, registry API, `test_all` diagnostic, and confirmation that callback errors are caught.
13. `README.md` API section updated to link the new `docs/api/rule.md` page.

### Out of scope

- **Event-driven triggers.** Rules are pollers. Event-driven action belongs in `sms.events.on` / `group:connect`. This is a deliberate design constraint — see Decisions D1 and the Section 4 "Out of scope" of the brainstorming transcript.
- **Action arguments (`...`).** Use closures. Lua makes this trivial.
- **Delay before the action.** Use `sms.timer.after` inside the action if needed.
- **Rule chaining / dependencies.** A rule's action can construct other rules; that's enough.
- **Persistence across mission restart.** Out of scope.
- **`EDGE_FALLING` / `EDGE_BOTH` / `N_TIMES` types.** Architecturally additive; can land later without breaking existing rules.
- **`:set_condition` / `:set_action` mutators.** If you need to swap the body, build a new rule with the same name (replacement is supported).
- **PyCharm / VS Code snippets to scaffold rule skeletons.** Out of repo scope.

## Constraints

- **Failure model: log + nil, never throw.** Per [`AGENTS.md` §3](../../../framework/AGENTS.md#3-failure-model-log--nil-never-throw). Bad opts → `log.warn` + return nil at construction time. Throws inside `condition` / `dev_condition` / `action` → `pcall`-caught → `log.error` + state machine treats the throw as "no fire this tick" (a thrown action does NOT unregister a `ONCE` rule — the next tick re-checks).
- **Lua 5.1 compatibility.** Mission environment.
- **Sim-time semantics.** All intervals and cooldowns are sim-seconds. Pauses with DCS, accelerates with time-compression. Inherited from `sms.timer`.
- **No inheritance, no metatable tricks beyond the universal handle pattern.** The handle is `setmetatable({...}, {__index = sms.rule})` like every other entity module. The `sms.rule` module table is itself made callable via `setmetatable(sms.rule, {__call = ...})` (custom — does not use `sms._make_callable_handle` because rule construction takes opts and registers, not just an existence check).
- **Idempotent module load.** Reloading `rule.lua` does NOT clobber the existing registry. Already-registered rules keep firing across a hot reload of the module file. (Achieved by guarding the registry table behind `sms.rule._rules = sms.rule._rules or {}`.)
- **One symbol, one home.** Per [`AGENTS.md` §6](../../../framework/AGENTS.md#6-loading-order). Everything `sms.rule.*` is defined in `framework/rule.lua`.

## Decisions

These are settled. Recording so future-me / future-agents don't relitigate.

### D1 — Pollers, not event handlers
Rules use `sms.timer.every`. They do not subscribe to `sms.events`. Event-driven action belongs in `sms.events.on` directly. Rationale: keeps the rule abstraction focused on stateful conditional logic; events are already a cleanly separated concern in dcs-sms. (Brainstormed Interpretation 1 vs 2; user picked "rules are rules, events are events".)

### D2 — Per-rule timer, not a shared manager scheduler
Each rule gets its own `sms.timer.every(interval, check)` handle on construction. No central tick loop. The "manager" in MOOSE becomes a passive registry. Rationale: dcs-sms already has the per-handle drift-corrected primitive; a shared scheduler would re-implement what `sms.timer` already does.

### D3 — Three types as enum strings, not metatable subclasses
`sms.rule.TYPE = { ONCE = "once", CONTINUOUS = "continuous", TOGGLE = "toggle" }`. The opts `type` field accepts the enum values (lowercase strings). No bare-string convenience — strict enum membership. Rationale: user explicitly wanted enum-style discoverability ("let's make an enum"); the lowercase string values keep log lines readable (`[sms.rule] helicopter_height (once) fired`).

### D4 — `cooldown` and `sustain` are orthogonal opts fields
Not new types. `cooldown = N` sec, default `0` (disabled). `sustain = N` sec, default `0` (disabled). Both compose with all three types except: `cooldown` is rejected on `ONCE` at construction (a one-shot rule has no second fire to gate). Rationale: keeps the type axis pure ("when does it fire relative to transitions?") and the gating axes orthogonal.

### D5 — `dev_condition` bypasses sustain AND cooldown
The dev path is the testing escape hatch. Setting `MIZ.helicopter_height = true` fires the action immediately and (for `CONTINUOUS`) repeatedly, regardless of cooldown or sustain. Rationale: the whole point of `dev_condition` is to short-circuit reality. Sustain/cooldown working against you in dev makes it useless.

### D6 — Auto-register on construction; explicit `:stop()` to disable
No `:add()` call — the MOOSE pattern of `RULE:New(...)…:Add()` is dropped. The constructor registers and starts the timer. To disable, call `:stop()` (which also unregisters). Rationale: "`:add()` after every constructor" is ceremony that catches users who forget; the auto-register matches dcs-sms's other constructors (`sms.timer.every` starts firing immediately, `sms.group.create` spawns immediately).

### D7 — Name collision replaces the old rule
Constructing a new rule with a name already in the registry causes the old rule to be `:stop()`-ed and replaced by the new one, logged at `info`. Rationale: matches DCS's same-named-entity behavior and dcs-sms's spawn auto-suffix philosophy of "the latest construction wins". Lets users hot-reload mission scripts without leaking stale rule timers.

### D8 — `:reset()` clears every per-rule state bit
`:reset()` clears the TOGGLE `active` flag, the in-flight `sustain_start`, and the `last_fire_time`. Useful for "the player restarted the level — re-arm every rule". Single method to clear everything, rather than separate `:reset_toggle()` / `:reset_cooldown()` / etc.

### D9 — `r:fire()` bypasses condition, cooldown, sustain
Manual fire is unconditional. Logged at `info` (`[sms.rule] helicopter_height: manual fire`). Counts as a real fire for `ONCE` (rule is unregistered), and updates `last_fire_time` so a subsequent natural fire still respects cooldown. Rationale: matches the developer mental model of "I want to verify the action runs right now".

### D10 — `test_all()` does not affect rule state
Diagnostic only. `pcall`s every condition, dev_condition, and action across all registered rules; logs PASS/FAIL per call. Does NOT update `last_fire_time`, does NOT flip TOGGLE `active`, does NOT unregister `ONCE` rules. The MOOSE `RULE_MANAGER:TestAllRules` semantics carry over. Rationale: testing should be safely repeatable.

### D11 — Module name is singular `sms.rule`
Matches `sms.timer`, `sms.group`, `sms.unit`. Even though there are many rules, the module follows the existing singular convention.

## Module structure (rough)

```lua
-- framework/rule.lua
assert(type(sms) == "table", "framework/sms.lua must be loaded first")
assert(type(sms.log) == "table", "framework/log.lua must be loaded first")
assert(type(sms.timer) == "table", "framework/timer.lua must be loaded first")

local log = sms.log.module("sms.rule")

sms.rule = sms.rule or {}
sms.rule._rules  = sms.rule._rules  or {}   -- name -> handle, registry
sms.rule._order  = sms.rule._order  or {}   -- ordered list of names for stable iteration

sms.rule.TYPE = {
  ONCE       = "once",
  CONTINUOUS = "continuous",
  TOGGLE     = "toggle",
}

local _handle_mt = { __index = sms.rule }

local function _is_handle(h) ... end

-- Validates opts; on success creates handle, registers, starts timer, returns handle.
-- On failure logs warn and returns nil.
local function _construct(name, opts) ... end

setmetatable(sms.rule, { __call = function(_, name, opts) return _construct(name, opts) end })

-- Handle methods (each accepts handle | name string for module-style equivalence).
sms.rule.get_name      = function(h) ... end
sms.rule.get_type      = function(h) ... end
sms.rule.is_active     = function(h) ... end
sms.rule.fire          = function(h) ... end
sms.rule.stop          = function(h) ... end
sms.rule.reset         = function(h) ... end
sms.rule.set_interval  = function(h, sec) ... end
sms.rule.set_cooldown  = function(h, sec) ... end
sms.rule.set_sustain   = function(h, sec) ... end

-- Registry.
sms.rule.get       = function(name) ... end
sms.rule.all       = function() ... end
sms.rule.remove    = function(name) ... end
sms.rule.test_all  = function() ... end
```

## State machine (canonical)

For each rule, on every interval tick:

```
v_real = pcall(condition);  if pcall failed, treat as false
v_dev  = pcall(dev_condition); if pcall failed, treat as false

-- dev path bypasses sustain + cooldown
if v_dev:
  _do_fire(via_dev = true)
else:
  -- real path: apply sustain, then state machine, then cooldown gating
  if v_real:
    if sustain_start == nil: sustain_start = now()
    effective = (now() - sustain_start) >= sustain
  else:
    sustain_start = nil
    effective = false

  if type == ONCE:
    if effective and cooldown_ok(): _do_fire(); _unregister()
  elif type == CONTINUOUS:
    if effective and cooldown_ok(): _do_fire()
  elif type == TOGGLE:
    if effective and not active and cooldown_ok():
      _do_fire(); active = true
    if not effective:
      active = false   # rearm

cooldown_ok() = (cooldown == 0) or (now() - last_fire_time) >= cooldown
```

`_do_fire(via_dev, via_manual)`:
- pcalls `action`
- on throw: log error; do NOT update `last_fire_time`; do NOT unregister even if `type == ONCE` (so a subsequent tick can retry). The throw wins over every other rule.
- on success: update `last_fire_time = now()`. For `type == ONCE`, unregister regardless of whether the cause was natural, dev, or manual.

Manual `:fire()` runs through `_do_fire(via_dev = true, via_manual = true)`. Logged at info (`[sms.rule] <name>: manual fire`). Same throw-handling rules — a thrown manual fire on a ONCE rule does NOT unregister it.

## Validation table for opts

| Field | Required | Allowed values | On bad input |
|---|---|---|---|
| `name` (1st arg) | yes | non-empty string | `log.warn` + return nil |
| `opts` (2nd arg) | yes | table | `log.warn` + return nil |
| `opts.type` | yes | one of `sms.rule.TYPE.*` | `log.warn` + return nil |
| `opts.condition` | yes | function | `log.warn` + return nil |
| `opts.action` | yes | function | `log.warn` + return nil |
| `opts.interval` | no (default `1`) | number `> 0` | `log.warn` + return nil |
| `opts.cooldown` | no (default `0`) | number `>= 0`; rejected if `> 0` and `type == ONCE` | `log.warn` + return nil |
| `opts.sustain` | no (default `0`) | number `>= 0` | `log.warn` + return nil |
| `opts.dev_condition` | no (default no-op `function() end`) | function | `log.warn` + return nil |

Mutators (`:set_interval`, `:set_cooldown`, `:set_sustain`) apply the same per-field validation. `:set_cooldown(N>0)` on a `ONCE` rule logs warn + returns false.
