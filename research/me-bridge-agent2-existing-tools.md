# Existing tools for external Lua execution against DCS — survey

Question: are there existing tools that already let an external process drive Lua execution against DCS, and in particular against the Mission Editor (ME) or main menu (i.e. *outside a running mission*)?

**Short answer: no.** Every tool I could verify operates inside a running mission. The closest precedents (`dcs_code_injector`, Quaggles `DCS.Lua.Connector`, DCS-gRPC, DCS-BIOS Lua Console, witchcraft) all share the same architectural shape — a `Scripts/Hooks/*.lua` script that drives its message pump from `onSimulationFrame()`, which only fires in-mission. None of them touch `MissionEditor.lua` or anything in `MissionEditor/modules/`.

---

## Per-tool findings

### 1. dcs_code_injector — Niels Vaes (the user's prior project)

- **What:** Python-based REPL inspired by Maya's script editor; lets you fire snippets at a running DCS mission.
- **Transport:** Hybrid. The Python app tails `dcs.log` for output and ships code into DCS via the hook (the README is light on the exact ingress mechanism — most likely a TCP socket bound by the hook, but I did not confirm against source).
- **Lua endpoint:** `Scripts/Hooks/dcs-code-injector-hook.lua` — GUI/hook env. Forwards code into the mission via `a_do_script(...)` (the same indirection dcs-sms's existing hook uses).
- **Active when:** running mission only. The README explicitly says "while a mission is running". No ME or menu support.
- **Source:** https://github.com/nielsvaes/dcs_code_injector

### 2. Quaggles — DCS.Lua.Connector

This is the most interesting precedent for the dcs-sms ME use case, because of one subtle feature.

- **What:** .NET library + console front-end + Lua hook. Send/receive Lua against a running DCS Lua environment.
- **Transport:** **UDP**. Hook binds `127.0.0.1:5000` (recv) and replies on `5001`, JSON messages.
- **Lua endpoint:** `Scripts/Hooks/DCS-LuaConnector-hook.lua`.
- **Multi-environment dispatch:** the hook accepts a `luaEnv` parameter on each request. When `luaEnv == "gui"` it uses `loadstring(code)()` to execute *directly in the hook env* (which has full file I/O, `net.*`, etc.); for other values it forwards via `net.dostring_in(env, code)`. This is the design pattern dcs-sms would want for an ME bridge — separate the transport from the target env, and target the GUI env when there is no mission.
- **Active when:** **mission only.** Critical caveat: the message pump is driven from `onSimulationFrame()`, which DCS only invokes during simulation. So even though the hook *can* run code in the GUI env, the listener is dormant in the main menu and the ME. The socket is bound at hook-load time, but nobody is calling `:receive()` until you start a mission.
- **Source:** https://github.com/Quaggles/DCS.Lua.Connector
- **Hook source:** https://github.com/Quaggles/DCS.Lua.Connector/blob/master/DCS-LuaConnector-hook.lua

This is the central finding of this survey: the GUI-env execution capability the user wants is already a known pattern, but the *driving callback* in every existing hook is `onSimulationFrame`, which kills the ME use case. dcs-sms either needs a different scheduling mechanism (ME-mod side, see below) or a callback that fires outside simulation.

### 3. DCS-gRPC (rust-server)

- **What:** Rust gRPC server that exposes DCS world state and accepts commands. Heavy, structured, schema-typed; the opposite design philosophy from the user's "lightweight" preference.
- **Transport:** gRPC over TCP, default `127.0.0.1:50051`.
- **Lua endpoint:** `Scripts/Hooks/DCS-gRPC.lua` plus a `MissionScripting.lua` patch and per-mission `GRPC.load()` (or autostart from the hook). Also requires modules under `lua/DCS-gRPC/`.
- **Active when:** running mission only. The README's two modes (`autostart=true` from the hook, vs. mission-driven `GRPC.load()`) both still depend on the simulation being live; the hook's per-tick scheduler runs on simulation frames. Issue #32 ("inject gRPC into all missions") shows the project is aware mission-only is a limitation.
- **Source:** https://github.com/DCS-gRPC/rust-server

### 4. DCS-BIOS Lua Console

- **What:** Web UI that submits Lua snippets for execution in any of four environments: `gui`, `export`, `mission`, or `hub` (the DCS-BIOS hub process itself).
- **Transport:** Hub is a separate process; web UI talks to hub, hub talks to DCS via DCS-BIOS's existing hook + Export.lua infrastructure (UDP/TCP — DCS-BIOS uses both for the cockpit data path; I did not chase down which carries Lua-Console traffic).
- **Lua endpoint:** the standard DCS-BIOS hook in `Scripts/Hooks/`.
- **Active when:** mission only — the docs say "Status indicators must show green connectivity before executing code", and the hook polling only happens during simulation. The "mission" env target is "the code snippets that are generated by the DCS: World mission editor [export] live in" (i.e. the runtime mission scripting state, not the ME UI itself).
- **Source:** https://dcs-bios.readthedocs.io/en/latest/lua-console.html

### 5. dcs-witchcraft (jboecker)

- **What:** Node.js server + browser console for inspecting / poking a running mission. Predates most of the others.
- **Transport:** TCP, mission scripting env opens a socket to the node server.
- **Lua endpoint:** Modifies `MissionScripting.lua` directly (calls `witchcraft.start(_G)` before sanitization). Not a `Hooks/` script.
- **Active when:** mission only — the bootstrap line runs as part of mission scripting init, so until a mission loads there is no client.
- **Source:** https://github.com/jboecker/dcs-witchcraft

### 6. DCSServerBot

- **What:** Discord-driven server-management bot. Runs missions on a schedule, manages slots, posts events.
- **Transport:** TCP between the Python bot and a hook script (default port `6666` per server instance).
- **Lua endpoint:** `Scripts/Hooks/`, auto-installed by the bot.
- **Active when:** the *bot* is always on (it manages the DCS dedicated server process from outside), but its in-DCS Lua reach is strictly mission-scope. README's mission-scripting examples (`do_script`, `do_script_file`, `dcsbot.sendBotMessage`) are all mission-side. No ME, no menu.
- **Source:** https://github.com/Special-K-s-Flightsim-Bots/DCSServerBot

### 7. DCS Olympus

- **What:** Browser-based real-time game-master / mission controller. Spawn units, task AI, etc.
- **Transport:** REST API to a webserver exe; in-DCS side is a DLL backend ("a bit like SRS does" per the README).
- **Active when:** running mission — Olympus is fundamentally a tool for *playing* the mission, not editing it. Spawning units, tasking AI, etc., all require the simulation to be live.
- **Source:** https://github.com/Pax1601/DCSOlympus

### 8. DCS Web Editor

Notable because it *does* expose ME-style editing externally — but it does so by parsing `.miz` files in the browser, not by talking to running DCS. To get its 3D view it instructs the user to open the .miz, save, run as mission (Ctrl+P), then have the Web Editor connect to the running mission. So it confirms the gap: even a project explicitly aimed at ME-like editing falls back to "run the mission" when it needs to talk to DCS state.

- **Source:** https://github.com/DCS-Web-Editor

### 9. dcs-master, dcs-http, Peksa Export.lua, etc.

Cockpit/instrument data exporters using `Export.lua`. Outbound only (or read-mostly), and Export.lua only runs when an aircraft is loaded — strictly in-mission.

### 10. DCS-Lua-Injector (VS Code extension, dzsekeb)

VS Code extension that ships Lua to a running mission. Mentioned for completeness; same in-mission constraint.

## What about callbacks that fire outside simulation?

The DCS GUI-environment callback list (`DCS.setUserCallbacks`) covers `onMissionLoadBegin/End`, `onSimulationStart/Stop/Frame/Pause/Resume`, network/player events, and `onGameEvent`. **There is no documented `onShowMissionEditor`, `onMenuFrame`, or anything that fires while the user is sitting on the main menu or in the ME.** I checked Hoggit's wiki and the ED API docs reference (`APIDCS_ControlAPI.html`); none surfaced.

That means: if dcs-sms wants a hook-driven listener that pumps in the ME, it will probably need either (a) a `coroutine`/timer trick that DCS keeps alive between sim sessions (I have no evidence anyone has gotten this to work), or (b) a separate listener inside the ME-mod (`MissionEditor/modules/dcs_sms_me/`), where the existing prefab-manager already runs and *does* have a render loop while the ME is open. The ME-mod path is the unexplored one — none of the surveyed projects touch `MissionEditor/modules/` at all.

---

## Tools that already solve this problem

**None.** No project I found provides external-driven Lua execution against the DCS Mission Editor's editable mission. The architectural pattern simply hasn't been built — every existing tool keys off the simulation lifecycle. The user is in greenfield territory for the ME bridge, and the dcs-sms ME-mod (which already has a foothold inside the ME GUI process) is plausibly the right place to host a listener that an external Claude can talk to.

## Things I couldn't verify

- Exact ingress transport for `dcs_code_injector` — README documents log-tail egress and a hook file, but I didn't read the source to confirm whether ingress is a socket, a file watcher, or `dostring_in` driven by the log somehow.
- DCS Olympus DLL details — README is deliberately light; "a DLL run by DCS" doesn't tell me which Lua state it injects into or whether anything could in principle run in the ME.
- DCS-BIOS Lua Console transport for the `gui`/`mission` envs specifically — I know DCS-BIOS uses UDP for cockpit data and a TCP control channel; I did not confirm which the Lua Console rides on.
- Whether anyone has experimentally run a long-lived `coroutine` from a hook that survives the no-mission state. Not seen in any surveyed project.
